# GraphSpec Issue 原则

这份 reference 把仓库里的 IssueDriven 规则压成一份短清单，供创建开发 issue 时快速对照。

## 什么叫合格的开发 issue

一个合格的开发 issue，本质上是一个“可收敛的不确定性”：

- 小到可以在一轮或几轮对话里收住
- 重要到如果继续模糊，后续工作就会一起漂
- 清楚到后续 spec 可以继续收紧，而不是从零猜起

它不是：

- 功能大礼包
- 纯施工顺序清单
- 没有背景的占位条目

## 必要段落，以及它们为什么存在

- `背景 / 为什么现在做`
  - 解释这个问题为什么现在必须收
  - 点出阻塞链路或漂移风险
- `目标 / 非目标`
  - 防止范围蔓延
  - 防止 AI 偷偷把未来 phase 的内容带进来
- `验收口径`
  - 说清什么必须成立、什么明确不成立、什么算失败
- `未决点`
  - 声明哪些问题还没拍板
  - 阻止实现者自行脑补答案
- `子任务树`
  - 把 issue 拆成面向收敛的小任务
  - 目的是把路径讲清，不是模仿巨大的实现计划

## 快速 readiness 检查

只要下面任意一个答案还很虚，这个 issue 就不该标成 `status:ready`：

1. 为什么现在做？
2. 这一刀到底在收什么问题？
3. 什么内容明确不在这次范围内？
4. 什么结果才算做成？
5. 哪些点还没定，不能被擅自补完？

## 按 phase 递增的 Harness

- Phase 0
  - format、lint、typecheck、smoke
- Phase 1
  - 图资产校验、fixtures、CLI 行为校验、一致性检查
- Phase 2
  - graph edit 护栏、回滚、审计
- Phase 3
  - Skill 到 CLI 的契约稳定性与失败路径
- Phase 4
  - 只有交互复杂度真的上来后，再引入 Playwright 这类更重的 E2E

Harness 要跟着当前 phase 最真实的失败模式一起长，而不是为了看起来完整就一次性前置建设。
