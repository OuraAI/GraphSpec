# gh-issue-delivery 工作清单

这份 reference 用来约束 issue 执行流程，避免跳过规划模式、跳过提问、跳过 spec 审核，或者把任务写成线性施工表。

## 1. 规划模式检查

开始前先确认：

- 当前是否在代理的规划模式下
- 是否能够使用结构化提问机制
- 当前 issue 是否已经足够聚焦，可以进入问题收敛

如果不在规划模式：

- 直接停下
- 要求用户切到规划模式
- 不要用普通文本提问来替代

## 2. 第一轮提问建议

至少问 `2-3` 个问题，优先覆盖：

1. 这次先收哪一刀
2. 哪些内容明确不进入本轮
3. spec 审核通过前，允许推进到哪里

如果 issue 本身已经写得很细，也不要跳过提问；问题的价值在于把实现边界再次钉住。

## 3. spec 最小完成物

在任何实现前，至少要有一份可持久化、可审核的局部 spec，明确：

- 背景 / 为什么现在做
- 目标
- 非目标
- 验收口径
- 未决点
- 依赖关系
- 任务拆分

如果 issue 已挂在项目既有 spec 体系下，优先更新现有 artifact；不要平行再造一套新文档。

## 4. 人工审核门槛

spec 产出后必须暂停，等待用户明确批准。

批准前不允许：

- 写代码
- 偷偷补实现细节
- 把 tasks 当成已经批准的施工单
- 用聊天摘要替代可审核的 spec 完成物

## 5. 任务图写法

任务必须按依赖关系组织，而不是按时间顺序堆叠。

至少拆成三类：

- 串行阻塞项
- 可并行项
- 审核 / 集成点
- 任务的写入边界

一个简单的判断标准：

- 如果任务 B 的边界依赖任务 A 的结果，B 属于串行阻塞项
- 如果两个任务互不改同一批文件，也不会互相等待结果，就应该并行

## 6. 子 Agent 委派规则

满足下面条件时，只要环境支持，就必须显式拆出并行分支，并至少委派一个子任务：

- 子任务边界稳定
- 写集互不重叠
- 不需要把未定设计问题一起交给子 Agent 决定
- 本地当前步骤不会被它立即阻塞

如果没有委派，必须记录原因。

委派时必须写清：

- 任务目标
- 输入上下文
- 输出预期
- 允许修改的文件范围

避免：

- 两个子 Agent 改同一个文件
- 把关键设计判断下放给子 Agent
- 主线程和子 Agent 重复做同一件事

## 7. 验证原则

验证要跟当前风险走，不要超前建设。

- 先做最轻量的仓库秩序与静态检查，例如格式、lint、类型或最小 smoke
- 当契约和输入输出开始稳定后，再补单测、契约测试、fixture 或行为验证
- 当流程、副作用、回滚、审计变得重要时，再补流程级验证
- 只有交互复杂度真的值得时，再引入重型 UI E2E

如果当前风险还停留在契约和边界，就不要提前引入重型 E2E。
