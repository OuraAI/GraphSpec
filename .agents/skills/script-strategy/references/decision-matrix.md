# 脚本选型矩阵

这份 reference 用来快速判断一个仓库脚本应该落在哪种实现方式。

## 默认规则

- 默认：`python`
- 薄胶水：`shell`
- 强耦合仓库 TS 逻辑：`bun/ts`
- 例外语言：`ruby`

## 什么时候用 shell

适合：

- 命令非常短
- 只是把已有命令串起来
- 没有结构化数据解析
- 没有复杂状态管理

不适合：

- 解析 YAML / JSON / Markdown
- 批量改文件
- 复杂错误处理
- 跨平台兼容复杂

一个简单判断：

如果你已经开始想用很多 `sed` / `awk` / `grep` / `cut` 去维持结构，就该退出 shell 了。

## 什么时候用 Python

优先场景：

- 结构化文件处理
- 文本批处理
- 代码生成
- 文档生成
- 批量文件系统操作
- 需要长期维护的辅助工具

Python 的价值不是“更高级”，而是它在这类任务上最稳。

## 什么时候用 Bun/TypeScript

只有在下面场景才优先：

- 要直接 import 仓库已有 TS 代码
- 要复用现有类型
- 要直接运行仓库内部运行时逻辑
- 项目已经有明确的 Bun/TS 脚本基础设施，复用收益大于额外重量

如果只是做一个独立的内容处理脚本，通常不必强行用 Bun/TS。

## 什么时候不要继续加 Ruby

不要把 Ruby 当默认脚本语言。

只有这些情况下才考虑：

- 仓库已经存在稳定 Ruby 生态
- 某个现成 Ruby 工具链明显更省事
- 环境约束已经保证 Ruby 可用

如果只是“当前机器刚好有 Ruby”，这不是充分理由。

## 兜底原则

优先顺序不是“技术偏好”，而是：

1. 够不够简单
2. 稳不稳定
3. 好不好维护
4. 能不能复用仓库现有逻辑

默认情况下，可维护性优先于一时省事。
